# 文档编写规则

指导 LLM 编写项目文档的规范，目标是简洁、清晰、可执行。

## 核心原则

- **简洁扼要**：删一切可有可无的修饰；一段只说一件事，≤5 行
- **逻辑清晰**：按因果/流程/分类组织，避免堆砌和跳跃
- **不仅讲是什么**：要说明为什么、怎么做
- **结构清晰**：善用标题/列表/代码块；标题不超过三级（过深则拆文档）

## 结构模板

```
# 标题
## 摘要       # 100–200字，覆盖背景/视角/结论/意义；短文档可省略
## 背景/问题   # 解决什么问题、为何重要
## 核心内容   # 按逻辑展开
## 结论       # 关键结论或下一步
## 参考       # 引用与相关文档
```

长文档（≥3 章节）在摘要后插入独立一行的 `[TOC]`，由阅读器自动生成目录。

## 摘要示例

> 大模型推理成本是 AI 落地的核心瓶颈。本文从计算与内存拆解 Prefill/Decode 成本，并对比量化、连续批处理、投机解码的实际收益。结论：Decode 阶段的内存带宽是效率关键，连续批处理是当下性价比最高的优化手段。

## 格式要点

- **列表**：并列用 `-`，顺序/优先级用 `1.`；列表项本身简洁，细节缩进展开
- **代码块**：命令/配置/示例指定语言以启用高亮
- **句式**：说明原因→"因为…所以…"；对比→"相比A，B的优势在于…"；步骤→"首先…其次…最后…"；总结→"综上所述…"
- **作者标注**（文首一行）：AI 创作→`> 作者：AI`；改编→`> 原作者：xxx`；本人→`> 作者：Jiamuyu`

## 数理表达

- 优先用文字解释概念；能用一句话说清的因果关系不用公式
- 公式仅用于必须量化之处，附文字说明每部分含义
- 复杂过程用 ASCII 流程图辅助，不用纯数学推导
- 表格只保留有说明意义的典型参数，不堆砌全部超参数

  ```
  好：KV Cache 随上下文线性增长，70B 模型 1K→2.7GB，100K→270GB
  差：KV Cache = 2 × seq_len × d_k × n_layers × n_heads × bytes
  ```

## 维护

- 文档改动后，在文首标注 `> 更新于 YYYY-MM-DD`
- 重大重构后同步目录索引；删除前确认无外部引用
