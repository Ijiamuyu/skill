# 文档编写规则

通用文档编写规范，适用于项目中的所有文档。

## 核心原则

- **简洁扼要**：用最少的文字说清一件事，删掉一切可有可无的修饰
- **有逻辑**：按因果、流程、分类等逻辑线索组织内容，避免跳跃和堆砌
- **分析说明**：不仅陈述"是什么"，更要说明"为什么"和"怎么做"
- **结构清晰**：善用标题、列表、代码块等格式，让读者一眼把握层次

## 格式规范

### 标题层级

```
# 一级标题（文档标题）
## 二级标题（主要章节）
### 三级标题（小节）
```

- 不超过三级标题，过深则说明需要拆分文档
- 标题应直接概括本节内容，避免模糊表述

### 段落

- 一段只说一个主题
- 段落长度不超过 5～7 行，过长应拆分
- 段落间用空行分隔

### 列表

- 并列关系用无序列表 `-`
- 顺序或优先级关系用有序列表 `1.`
- 列表项本身简洁，细节缩进展开

```
- 核心概念：解释 A 是什么
  - 关键特性 1：xxx
  - 关键特性 2：xxx
- 工作原理：说明 A 如何运作
```

### 代码块

- 命令、配置、示例代码用代码块包裹
- 指定语言以启用语法高亮

## 文档结构

一篇好的文档应包含以下要素（视篇幅灵活取舍）：

```
# 标题

## 背景 / 问题
说明为什么要写这篇文档，解决什么问题

## 核心内容
按逻辑线索展开正文

## 总结 / 结论
回顾关键结论或下一步行动

## 参考
引用的资料、链接、相关文档
```

## 常用句式规范

| 目的 | 推荐句式 |
|------|---------|
| 说明原因 | 因为……所以…… |
| 对比选择 | 相比 A，B 的优势在于…… |
| 列出步骤 | 首先……其次……最后…… |
| 总结观点 | 综上所述…… |

## 修改规则

- 改动文档时，在文件顶部用 `> 更新于 YYYY-MM-DD` 或类似标记记录变更
- 重大重构（拆分/合并文档）后更新目录索引
- 删除内容前确认没有外部引用指向该内容
- 保持全文语气和术语风格一致

## 示例

```
# 定价模型分析

## 背景
AI 推理成本持续下降，需要一套可量化的定价模型指导服务定价。

## 核心内容
1. **成本构成**：算力 + 存储 + 网络 + 运维
2. **定价策略**：
   - 按 token 计费：单价 × 消耗量
   - 包月套餐：固定费用，用量上限
3. **对比分析**：相比按 token 计费，包月套餐更适合高频用户

## 结论
建议采用混合定价模式：小额用户按 token 计费，大客户走包月套餐。
