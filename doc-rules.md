# 文档编写规则

指导 LLM 编写项目文档的规范，目标是简洁、清晰、可执行。

## 核心原则

- **简洁扼要**：删一切可有可无的修饰；一段只说一件事，≤5 行
- **逻辑清晰**：按因果/流程/分类组织，避免堆砌和跳跃
- **不仅讲是什么**：要说明为什么、怎么做

## 结构模板

```
# 标题
## 摘要       # 100–200字，覆盖背景/视角/结论/意义
## 核心内容   # 按逻辑展开
## 结论       # 关键结论或下一步
```

长文档（≥3 章节）在摘要后插入 `[TOC]`。

## 格式要点

- 列表：并列用 `-`，顺序/优先级用 `1.`
- 代码块：指定语言以启用高亮
- 标题不超过三级，过深则拆文档
- 复杂流程用 ASCII 图，不用纯数学推导
- 不使用 `---` 水平分割线 —— 标题层级已足够分隔内容，无额外价值

## 作者标注

仅知识/总结类文档在文首标注一行：

- AI 创作 → `> 作者：AI`
- 改编 → `> 原作者：xxx`
- 本人 → `> 作者：Jiamuyu`

skill 文件不标注（无助于规范本身）。

## Skill 文件规范

- **命名**：英文小写 + 连字符 + `-rules.md` 后缀（如 `chat-rules.md`）
- **存放**：放到独立 git 仓库（路径不限），业务仓库通过相对路径引用，不复制副本
- **职责单一**：一份 skill 只解决一类任务
- **抽象层级**：保留方法论骨架（原则 + 流程 + 边界）；案例、示例由调用方文档承载
- **本质**：任务是规范/约束，不是具体执行动作

## 维护

- 知识/总结类文档改动后在文首更新 `> 更新于 YYYY-MM-DD`
